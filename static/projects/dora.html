30-Second Overview

**Codeilus** is a Rust-powered tool that takes any codebase and turns it into a **gamified, interactive learning experience** you can explore in your browser. Think of it like Duolingo, but instead of learning Spanish, you're learning how a software project works — its architecture, key symbols, how modules connect, and why things are built the way they are. It parses source code, builds a knowledge graph of symbols and relationships, generates narratives and quizzes, and serves it all through a web UI with progress tracking and badges. It's built for **newcomers joining a team**, open-source contributors trying to understand an unfamiliar repo, or anyone who's ever stared at a large codebase thinking "where do I even start?" Rather than reading docs (if they exist) or grep-ing aimlessly, Codeilus creates a structured curriculum — chapters ordered by difficulty, community-detected module clusters, and LLM-generated explanations — so you can learn a codebase the way you'd learn a subject in school. Under the hood, it's a **modular Rust workspace of ~16 crates**: `codeilus-parse` extracts symbols and imports from source files, `codeilus-graph` builds the dependency/call graph, `codeilus-analyze` detects communities and entry points (like the 23 communities and 1,048 symbols you see above), `codeilus-narrate` and `codeilus-llm` generate human-readable explanations, `codeilus-learn` handles the gamification layer (quizzes, badges, progress), and `codeilus-api` serves it all via an Axum web server. Everything flows through an event bus and SQLite database, keeping the architecture clean and each piece independently testable. It's currently past Sprint 0 (compiles, tests pass, zero warnings) and heading into the feature waves that bring each crate to life.

How It Works

## Codeilus Architecture: A Guided Tour **The Front Door and the Post Office.** Every request into Codeilus enters through one of two `main` entry points — the CLI binary and the web server. Think of the **router/API layer** (community `router_list`, `response_routes_types`) as the post office: it receives incoming HTTP requests, figures out which department should handle them, and routes them accordingly. The `router()` functions (there are several, one per domain) assemble Axum route trees for files, symbols, communities, chapters, and more. Types like `HealthResponse`, `ChapterResponse`, and `SearchResult` define the envelopes those responses travel in. The `AppCache` acts as a shared mailroom, holding pre-computed data so the post office doesn't have to re-fetch everything on every request. **Parsing and Understanding the Code.** When Codeilus ingests a repository, the **parse pipeline** (communities `parse_extracts`, `fetch_parse`, `resolve_tree`) does the heavy lifting. Think of it as the intake department at a hospital — it triages every source file, extracts symbols (819 functions, 165 structs, 24 interfaces, etc.), resolves imports, and classifies each symbol's role. The `resolve_tree` community is the largest working group here (121 members), handling everything from `parse_repository_with_tempdir` to `classify_symbol` and `score_entry_points`. It builds the foundational data — parsed files, symbol metadata, dependency edges — that every downstream module relies on. The `file_reader_interfaces` and `read` micro-communities are its humble helpers, abstracting raw file I/O behind a clean `Reader`/`FileReader` trait. **Storage, Search, and the Data Backbone.** Once code is parsed, the **repository/database layer** (communities `list_insert`, `list_empty`, `insert_file`, `insert_snippet`, `resolve_fingerprint`) persists everything into SQLite through structured repos — `FileRepo`, `SymbolRepo`, `PipelineRepo`, and friends. Think of this layer as the filing cabinet system: `insert_file` handles storing and retrieving files, `insert_snippet` manages code excerpts with context windows, and `list_insert` powers queries like "list hotspots" or "search unified." The `resolve_fingerprint` community adds cross-cutting concerns — error types (`CodeilusError`), security patterns, migration management, and configuration defaults. Together, these modules ensure that every piece of extracted knowledge is stored once and queryable from anywhere, honoring the architecture rule that all DB operations flow through repository structs. **Intelligence: Graphs, Metrics, Diagrams, and Narratives.** The analytical brain of Codeilus lives in several specialized communities. The **graph and metrics** modules (`generate_mermaid`, `escape_flowchart`, `batched_placeholder`) transform raw symbol relationships into visual diagrams — Mermaid flowcharts, architecture graphs, community visualizations — with careful label escaping and node-limit controls. The **LLM and narrative** layer (`prompt_accumulator`, `row_curriculum`, `chapter_record`) is like the storytelling department: it takes the structured analysis and, using prompt templates and an `AnthropicApiStub`/`NoOpProvider` abstraction, generates human-readable chapters, quizzes, and heritage questions. The `EventBus` in `row_curriculum` acts as the internal PA system — broadcasting progress updates so that the UI, the narrator, and the learning engine all stay in sync without directly coupling to each other. **The Learning Experience.** At the outermost ring, the **curriculum and gamification** modules (`curriculum_chapter`, `chapter_record`, `index`) assemble everything into an interactive learning journey. `curriculum_chapter` estimates difficulty, orders content for reading, and checks badge criteria. `chapter_record` tracks learner progress — quizzes passed, sections completed, exploration recorded — and awards badges like "Chapter Champion." The tiny `index` community generates the landing page that lists available repositories. Data flows in one direction: raw code → parsed symbols → stored relationships → computed metrics/graphs → LLM-generated narratives → curriculum chapters → learner progress. This clean pipeline, glued together by the `EventBus` and shared `core` types, is what lets Codeilus turn any codebase into a gamified course with a single command.

Why Trending

Here's why developers are buzzing about **Codeilus**: **Codeilus is attacking one of the most underserved pain points in modern software engineering: the comprehension gap.** AI tools have supercharged how fast we *write* code, but understanding unfamiliar codebases is still a brutal slog of grepping, guessing, and asking whoever's been around longest. Codeilus flips the script entirely — you point a single Rust binary at any repo, and it extracts the full structural fingerprint (symbols, relationships, dependency communities, metrics), then transforms it into a gamified, browser-based learning experience with chapters, quizzes, architecture diagrams, and LLM-narrated explanations. The numbers from this analysis alone tell the story: **1,048 symbols, 3,718 relationships, and 23 automatically detected communities** — that's not a toy demo, that's a real codebase being decomposed into navigable, teachable units with cohesion scores and ranked entry points. What's *really* turning heads, though, is the ambition of the two-mode design and the engineering rigor behind it. Locally, you get a full SvelteKit 5 interactive explorer with streaming Q&A powered by Claude, impact analysis ("what breaks if I touch this function?"), and complexity heatmaps. But the killer feature might be **static grasp pages** — self-contained HTML files under 500KB that give you a "5-minute understanding" of any trending GitHub repo, deployable to a CDN via cron. The whole thing is built as 16 purpose-separated Rust crates with strict dependency rules (`codeilus-core` is the shared contract, zero cross-dependencies between siblings), event-driven architecture via `tokio` broadcast channels, and zero-tolerance for clippy warnings. It's the kind of project that makes you think, "Why didn't this exist already?" — and the fact that it's a single binary with no runtime dependencies beyond Claude Code makes the adoption story incredibly clean.
